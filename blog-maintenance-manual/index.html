一、 版本迭代命名规范 (Versioning Rules)

本规范旨在为博客的更新迭代建立一套清晰、可预测的版本管理体系。所有更新日志均须遵循此规则。

核心结构: 主版本号 + 更新类型标签 + 简短描述

示例: v3 ✨ Feature: 新增文章内容检索功能

1.1 主版本号 (Major Version - vX)

主版本号代表博客的重大里程碑。它的变更意味着底层架构、核心设计或整体方向发生了根本性变化。

  • 升级时机 (When to Upgrade):
    • 技术栈迁移: 例如，从 GitHub Pages 迁移至 Cloudflare Pages (即 v4 的核心任务)。
    • 主题或框架重构: 例如，从 Hugo 的某个主题彻底更换为另一个，或自己从零开始重写主题。
    • 内容方向大改: 例如，博客从个人生活记录彻底转型为专业技术分享。
  • 当前版本: v3

1.2 更新类型标签 (Update Type Tag)

每个标签都配有专属的图标和颜色，用于在更新日志中快速识别更新的性质。

图标 英文标签 中文含义 适用场景 示例
✨ Feature 新功能 增加用户可感知的新模块或新能力。 新增文章内容检索、增加评论系统
🎨 Style 样式 不影响功能的视觉调整，如颜色、字体、布局。 优化暗色模式配色、调整分割线样式
🐛 Fix 问题修复 修正已知的 Bug 或错误。 修复小屏幕目录显示错位
📝 Docs 文档 撰写或更新说明性文档，如本手册。 创建博客维护手册
🚀 Perf 性能优化 提升网站加载速度或运行效率。 优化图片加载速度
♻️ Refactor 代码重构 未增加新功能或修复问题的代码结构调整。 重构短代码逻辑
🧹 Chore 杂务 构建过程、工具链或依赖项的更新。 升级 Hugo 版本、清理无用文件

二、 博客建设指导规则 (Construction Guidelines)

这些规则旨在确保博客内容和功能开发的质量与一致性。

2.1 开发流程 (Development Workflow)

  1. 规划 (Plan):
    • 任何非紧急的修复或调整，都应先在 “迭代蓝图” (即你的待办清单) 中进行记录和规划。
    • 明确每次迭代的核心目标 (例如，v3 期间完成搜索功能)。
  2. 执行 (Execute):
    • 遵循上文的 版本迭代命名规范 来记录每一次变更。
    • 在本地进行充分测试，确保新功能或修复在不同设备 (PC/移动端) 和模式 (亮色/暗色) 下均表现正常。
  3. 发布 (Deploy):
    • 完成测试后，将代码推送到主分支并部署上线。
    • 在博客的 “更新日志” 页面发布对应的版本更新说明。
